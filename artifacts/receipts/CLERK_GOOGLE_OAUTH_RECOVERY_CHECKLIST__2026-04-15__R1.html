ThetaFrame / Clerk / Google OAuth

Recovery Checklist

This is the exact checklist to get Google sign-in working again for ThetaFrame production. Repo-side blockers have already been removed; what remains is Clerk and Google Cloud configuration plus one production verification pass.

Important: your Gmail browser session is not the OAuth configuration itself. Google sign-in only comes back online when Clerk has a working Google social connection and Google Cloud has the matching OAuth client, domains, and redirect URI.

What I already fixed in ThetaFrame

  • Clerk social buttons are no longer hidden by app CSS.
  • The sign-in and sign-up pages no longer claim that Google is disabled.
  • The repo’s expected Clerk proxy path is /api/__clerk.

After the next deploy/build, the Google button should appear automatically if Clerk is correctly configured.

  • App domain: https://thetaframe.mrksylvstr.com
  • Clerk proxy path: /api/__clerk
  • Expected proxy URL: https://thetaframe.mrksylvstr.com/api/__clerk

Step 1: Fix Clerk dashboard settings

  1. Open the correct production Clerk application for ThetaFrame.
  2. Go to SSO connections / Social connections.
  3. Open the Google connection.
  4. Ensure the Google connection is enabled for production sign-in.
  5. If Clerk shows a generated redirect URI, copy it exactly. You will use that exact URI in Google Cloud.
  6. If Clerk offers “use your own credentials” or custom Google OAuth credentials, turn that on for production and be ready to paste the Google client ID and secret from Step 3.

Step 2: Fix Clerk domains and proxy

  1. In Clerk, go to Domains.
  2. Make sure thetaframe.mrksylvstr.com is registered as a production domain.
  3. Set the Clerk proxy URL to: 
    https://thetaframe.mrksylvstr.com/api/__clerk
  4. Confirm ThetaFrame production also has: 
    VITE_CLERK_PROXY_URL=https://thetaframe.mrksylvstr.com/api/__clerk

Step 3: Fix Google Cloud OAuth credentials

  1. Open the correct Google Cloud project for ThetaFrame auth.
  2. Go to Google Auth Platform / Branding and confirm the app branding/consent screen is configured for production use.
  3. Go to Google Auth Platform / Clients and create or edit a Web application OAuth client.
  4. Add the production JavaScript origin: 
    https://thetaframe.mrksylvstr.com
  5. Add the exact Clerk redirect URI from Step 1 under Authorized redirect URIs. Do not guess this value; paste it exactly.
  6. Copy the OAuth client ID and client secret.

Step 4: Connect Google Cloud credentials back into Clerk

  1. Return to the Google social connection in Clerk.
  2. Paste the Google client ID and client secret from Google Cloud.
  3. Save the connection.
  4. Ensure the Google connection is available in the same Clerk production app that ThetaFrame is using.

Step 5: Verify ThetaFrame production

  1. Load https://thetaframe.mrksylvstr.com/sign-in.
  2. Confirm the Google button appears in the Clerk sign-in component.
  3. Use Google sign-in and verify ThetaFrame lands on a signed-in lane shell.
  4. Then rerun the repo-native capture flow: 
    THETAFRAME_BROWSER_BASE_URL=https://thetaframe.mrksylvstr.com pnpm run qa:browser:auth:capture
  5. After that, capture admin state and rerun browser QA.

If Google still does not appear

  • Wrong Clerk app/environment
  • Google connection disabled in Clerk
  • Google OAuth client missing or using the wrong redirect URI
  • Clerk production domain / proxy mismatch
  • Deployment not updated with the current frontend changes

Repo-side UI suppression is already fixed. If the button is still absent after deploy, the remaining problem is Clerk or Google Cloud configuration, not ThetaFrame page code.
